## Project Overview

This is a minimal, from-scratch neural language model (MiniLM) written in pure NumPy—no frameworks. The goal is to build a real, learning AI system that progressively replaces hardcoded algorithms with learned behavior. The assistant speaks Hebrew and English, runs in a Tkinter GUI (chat.py), and integrates math, code generation, and knowledge lookup as deterministic tools triggered by the model's learned decisions.

**Core principle:** Intelligence lives in the *decision* to call a tool; the tool itself is deterministic. The model learns *what to do*, not how to compute it.

## Architecture

### Core Components

- **[model.py](model.py)** — The neural network core. A 2-layer feedforward architecture:
  - Embedding layer: each token becomes a dense vector (embed_dim, default 64)
  - Hidden layer: context_len * embed_dim → hidden_dim (default 128) via ReLU
  - Output: softmax probabilities over vocab (vocab_size)
  - Forward pass, cross-entropy loss, backprop (SGD), and parameter updates are all here
  - Minimal, readable, fully differentiable

- **[tokenizer.py](tokenizer.py)** — Word-level tokenizer with vocab management
  - `build_vocab()` scans training texts, builds bidirectional word↔index mappings
  - `encode()` / `decode()` handle text ↔ integers
  - Special tokens: `<PAD>`, `<UNK>`, `<BOS>`, `<EOS>`
  - Handles Hebrew text with diacritics, punctuation stripping, lowercasing

- **[train.py](train.py)** — Training loop with incremental learning
  - Loads all data sources (see below), tokenizes, runs SGD epochs
  - **Key feature:** hash-based change detection. Only trains on new/changed articles; skips unchanged ones
  - Saves `model.npz` (weights) and `tokenizer.json` (vocab); also saves `trained_hashes.json` for next run
  - Parallelizes nothing; ~14s/epoch on ~6.4k training samples (context_len=8)

- **[chat.py](chat.py)** — Tkinter GUI and inference loop
  - Single place for all UI logic
  - `send()` function: orchestrates `quick_tool_reply` (instant) → model generation → `parse_calc_call` (math) → else `knowledge_reply` (wiki/sources)
  - Handles RTL (Hebrew) text wrapping and font fallback
  - Stores conversation history; "Clear Chat" button resets

- **[assistant_tools.py](assistant_tools.py)** — External tools (deterministic)
  - `quick_tool_reply()`: Handles greetings, identity ("Who are you?"), coded-action triggers (code gen, source lookup)
  - `execute_calc()`: Arithmetic from model's `<calc> +/−/×/÷` signals; always exact (no ML guessing at math)
  - `parse_calc_call()`: Detects if model output starts with `<calc>`; extracts operator
  - `knowledge_reply()`: Wikipedia search (fallback if model can't answer locally), source document lookup, "I don't know" if nothing found

### Training Data Pipelines

Data is split into modular JSON files so new sources can be added independently:

- **[training_data.json](training_data.json)** — Hand-written dialogue and knowledge in Hebrew + English
- **[math_training_data.json](math_training_data.json)** — Arithmetic exemplars (generated by add_math_data.py)
  - Each sample: user asks "what is 7 plus 3" (in various Hebrew/English phrasings), model outputs `<calc> +`
  - Operands are varied (not always 7,3) so the operator is the only predictable signal
  - Model learns to recognize arithmetic intent across word variations: כפול, פעמים, פי, ועוד, פלוס, פחות, מינוס, חלקי, מחולק, הסכום של, ההפרש בין, המכפלה של
- **[better_code_examples.json](better_code_examples.json)** — Code generation exemplars
- **[simple_code.json](simple_code.json)** — Simple Python code snippets
- **[new_training_examples.json](new_training_examples.json)** — Miscellaneous new examples (append as needed)
- **API-fetched data** (optional; see collect_corpus.py):
  - `api_training_data.json`, `wikipedia_training_data.json`, `github_training_data.json`
  - Generated via `collect_corpus.py --source [wikipedia|github|both]`
  - Not in git (too large); rebuild locally if needed

**Training data format:** List of dicts with `{"id": "...", "text": "..."}`. The `id` must be stable across runs (hash-based skipping depends on it).

### Hyperparameters

Controlled via environment variables (train.py):
- `CONTEXT_LEN=8` (default) — how many prior tokens the model sees; raised from 5 to 8 to accommodate longer operation words like "הסכום של"
- `EMBED_DIM=64` (default) — embedding dimension
- `HIDDEN_DIM=128` (default) — hidden layer size
- `EPOCHS=120` (default) — training passes
- `LR=0.01` (default) — learning rate (SGD)
- `LOG_EVERY=10` (default) — print loss every N epochs

Example: `CONTEXT_LEN=10 EPOCHS=60 python train.py`

## Common Tasks

### Train the model
```bash
python train.py
```
- Detects if `training_data.json` and friends have changed (via hash)
- If no changes: instant (0 epochs, ~1s)
- If changes: trains only new/changed articles across the data sources, updates `model.npz` and `trained_hashes.json`

### Run the full app (train + GUI)
```bash
python run.py
```
- Calls `train()` first (quick if no changes), then opens chat GUI

### Run CLI mode (no GUI)
```bash
python ask.py "What is 2 plus 2?"
```
- Single query, prints response to stdout
- Useful for testing or scripting

### Add new training data
1. Edit the relevant JSON file or create a new one:
   - Dialogue/knowledge → append to `training_data.json`
   - Arithmetic → run `python add_math_data.py` to regenerate exemplars
   - Code → append to `better_code_examples.json` or `simple_code.json`
   - Domain-specific → create a new `.json` file, add path to `DATA_PATHS` in train.py

2. Run training:
   ```bash
   python train.py
   ```
   Hash mismatch triggers retraining on changed articles.

### Adjust hyperparameters
```bash
CONTEXT_LEN=12 HIDDEN_DIM=256 EPOCHS=200 LR=0.005 python train.py
```

### Fetch external data (Wikipedia, GitHub)
```bash
# Wikipedia only, save to api_training_data.json
python collect_corpus.py --source wikipedia --out api_training_data.json

# GitHub code samples
python collect_corpus.py --source github --github-limit 20

# Merge all sources into training_data.json (careful: this is slow and adds volume)
python collect_corpus.py --source both --merge-into training_data.json
```
Note: GitHub requires a `GITHUB_TOKEN` env var for higher rate limits. Skipped if not set.

## Design Decisions & Constraints

### Why no frameworks (NumPy only)?
- Educational transparency: every matrix multiply, every gradient is visible and auditable
- Memory/compute efficiency: no overhead from Keras, PyTorch, TF
- But trade-off: no GPU, no batching, single-threaded (14s/epoch is acceptable for this scale)

### Why word-level tokens, not character/BPE?
- Simpler pipeline; Hebrew punctuation + diacritics handled upfront in tokenizer.clean()
- Smaller vocab; fewer "unknowns"
- Context window (8 tokens) is still readable (∼1–2 sentences)
- Trade-off: OOV words become `<UNK>`; no subword generalization

### Why SGD, not Adam/RMSprop?
- Simpler; no state to track
- For small vocab/network, plain SGD works well
- Math training data shows strong convergence in 120 epochs

### Why deterministic tools for math/code/identity?
- Correctness: the model's decision ("this is math") can be *learned*; the execution must be *always right*
- Separation of concerns: intelligence (decision) vs. computation (tool)
- Extensible: new tools can be added without retraining if the model learns to emit the right signal

### Why incremental (hash-based) training?
- Fast iteration: adding one article doesn't retrain the whole corpus
- But: vocab changes (new word in data) force a full from-scratch retrain because embeddings are word-specific
- If vocab is stable, most re-runs are instant

### Why context_len=8?
- Raised from 5 to 8 to fit phrases like "הסכום של" (sum of, 3 tokens) while leaving room for operands
- Constraint for data: operands must appear in order (a then b); operation word inside the 8-token window

## Project Philosophy

**Goal:** Remove hardcoding; let the model learn.

**Current state (as of 2026-07-02):**
- **Math (done):** Model learned to emit `<calc> +` when it sees arithmetic intent. `execute_calc()` then interprets and computes. Covers 20+ operation phrasings in Hebrew and English.
- **Code generation (in progress):** `generate_code()` is still partially hardcoded template lookup. Next target: model learns to emit `<code>` signal; tool fills the template.
- **Knowledge routing (in progress):** Wikipedia/local source lookup is tool-driven. Next: model learns to emit `<wiki>` when it needs external data.
- **Greeting/identity (done):** Model trained on "I'm a Hebrew-English AI assistant built from scratch in NumPy."

**Future:** Remove hardcoded greetings → model learns. Remove code templates → model learns what to output. The vision is an emergent system where the network's learned *decisions* drive behavior, and tools handle *execution*.

## Troubleshooting

- **Model predictions are random:** Likely not trained yet or vocab size is very small. Check `model.npz` exists.
- **Training is slow:** Single-threaded NumPy on CPU. ~14s/epoch is normal. Use `LOG_EVERY=10` to reduce print overhead.
- **Hebrew text looks garbled in GUI:** Font fallback in chat.py should handle it; if not, check system font support for Hebrew.
- **Wikipedia lookups fail:** Rate-limited or no internet. Graceful fallback to "I don't know."
- **Vocab mismatches after retraining:** If you manually edited training JSON, hash might not detect changes. Delete `trained_hashes.json` to force full retrain.
